- **D-01:** Bump version to **2.2.0** (semver minor). Adding a new pre-built domain with external API enrichment is a new capability, not a patch. All version references update: `plugin.json`, `marketplace.json`, `CHANGELOG.md`.

### Artifact Scope
- **D-02:** Update **all four surfaces**: core metadata (plugin.json + marketplace.json), CHANGELOG, README pre-built domains list, and CLAUDE.md + DEVELOPER.md architecture/schema sections.
- **D-03:** Every surface that lists pre-built domains must include clinicaltrials as the 3rd entry alongside drug-discovery and contracts.

### Clinicaltrials Documentation Depth
- **D-04:** **Listed as 3rd domain only** — no new showcase doc in this phase. Add a 1-2 sentence description in README covering: 12 entity types, 10 relation types, CT.gov v2 + PubChem PUG REST enrichment via `--enrich` flag. No `docs/showcases/clinicaltrials.md` in this phase.

### ADDING-DOMAINS.md Enrichment Section
- **D-05:** Add a **full `enrich.py` section** to `docs/ADDING-DOMAINS.md` documenting the enrichment extension pattern as an optional addition to the standard domain package. Clinicaltrials is the worked example. The section should cover:
  - When to use enrichment (external APIs that patch graph nodes post-build)
  - The `enrich_graph(output_dir)` contract — reads `graph_data.json`, fetches APIs, writes `_enrichment_report.json`
  - The `--enrich` flag wiring in `commands/ingest.md` Step 5.5
  - The helper function pattern: `_fetch_*` functions return `None` on failure (non-blocking)
  - Reference `domains/clinicaltrials/enrich.py` as the canonical implementation

### Claude's Discretion
- Exact wording of clinicaltrials description in README and plugin.json — should match the domain's actual capabilities as implemented
- Which subsections of DEVELOPER.md Domain Schema Reference to update vs. which to leave as drug-discovery-focused
- Whether CHANGELOG 2.2.0 entry references the CTDM requirement IDs
